L2CP의 'C' 창연입니다!

개발자들이 팀을 이뤄서 프로젝트를 진행하고 관리할 때 대부분 git 을 사용한다.

우리 프로젝트도 마찬가지로 git 을 이용하여 진행하고자 했다.

해당 기술을 사용하는 이유는 효율적인 업무분배와 버전 관리를 위함이다.

예를 들어..

  • 서로의 개발 영역을 나눠서 파일 업로드

  • 최신의 작업 브랜치를 가져와 반영하면서 서로의 코드 conflict를 확인

  • 오류가 났을 땐 이전 버전으로 돌아가서 다시 작업

이런 작업을 진행할 수 있기에 git 을 사용하는 것이다.

그러나 위 내용이 git이라는 버전관리 도구 사용 을 통해서 문제를 해결하는 것이라면,

우리는 우리만의 git 브랜치 전략 을 통해 개발 방향성이나 브랜치 안정성 등을 고려한

프로젝트 흐름에 알맞는 방법을 선택하고자했다.

이에 대표적인 브랜치 전략을 찾아서 비교했다.

Git Flow 전략

  • main, develop 등 브랜치의 역할을 정해 체계적인 관리

  • 배포 역할의 브랜치에서 버그 관리를 도맡아서 진행

  • 그러나, 복잡한 브랜치 구성과 팀원들의 사전 지식이 필요

  • 느린 주기로 인한 배포 주기도 길어짐

Github Flow 전략

  • main(default) 브랜치를 항상 배포 가능한 상태로 유지

  • 변경 사항이 빠르게 main 브랜치에 통합

  • 지속적인 통합 CI & 자동화 환경에 유리


분석 내용에 대해서 우리 프로젝트에 알맞은 방식은 Git Flow 방식이라고 결론지었다.

우리의 프로젝트 팀원들이 여러 기능들을 나눠서 개발하기에 각각의 브랜치들이 명확한 역할을 가지고 있을 뿐더러, 최종적으로 main 브랜치를 기준으로 배포를 진행할 예정인데 오류 발생 시 main 브랜치에서 직접적으로 수정하는 방식은 매우 불안정하다고 생각했다.

또한 feature 브랜치를 배포를 진행하는 브랜치에 직접 합치는 방식은 Conflict를 유발하는 방식이라고 생각했기 때문이다.

협업을 위한 또 하나의 설정

또한 협업이다 보니 팀원들의 피드백을 활용하는 방식을 사용하고자 했다.

  • 내가 작성한 코드에 대한 객관적인 시선이 필요

  • 사전에 설정한 코드 컨벤션에 대한 확인

  • 좀 더 좋은 방식을 보완 받을 수 있음

이에 대한 이유로 각자 개발한 코드에 대해서 같이 프로젝트를 진행하는 팀원들에게 피드백을 받고 이에 대해서 승인을 받을 때 최종적으로 merge를 허가하는 Rule을 지정했다.

이렇게 된다면 팀원이 작성한 코드를 볼 때, 많은 내용이 있으면 리뷰를 하는 입장에서 이해하기 어렵고 코드 리뷰의 집중력이 떨어진다고 생각했다.

그래서 PR(Pull Request)의 크기를 줄이고자 다음과 같은 방법을 제안했다.

// 기존
develop <- Feature/#10-Login (PR 후 merge)

// 새로운 방식

Feature/#10-Login <- Feature/#10-Login-Layout (PR 후 merge)
Feature/#10-Login <- Feature/#10-Login-Feature (PR 후 merge)

...모든 기능이 완료된 이후
develop <- Feature/#10-Login

  1. 각자의 상위 기능 브랜치를 만들고

  1. 각 상위 기능 브랜치의 하위 브랜치를 생성하고

  2. 명시된 하위 기능 개발이 완료된 이후에 -> 자신의 상위 브랜치로 merge 진행

  3. 이후에 최종적으로 develop 브랜치에 합치기

이 방식은 팀원들도 긍정적으로 생각하여 결과적으로 프로젝트에 적용하기로 했었다.. 하지만..

한 가지 간과한 사실.. 바로 최신화

프로젝트 설정을 모두 마무리하고 각자 역할을 분배한 이후에 개발을 진행하고 있었다.

그러나 얼마 지나지 않아서 브랜치 전략이 실패했다는 것을 깨달았는데, 바로 연관된 기능에 대한 최신화 반영 이였다.

예시로 팀원 A, B를 상정해서 얘기를 해보자면

팀원 A가 로그인 기능을 개발 완료
Feature/Login <- Feature/Login-DB-연결
그러나 기본적인 UI 테스트를 위해서 Feature/Login-UI-Test 작업을 진행한 이후에 develop에 합칠 예정

이 때 팀원 B가 로그인과 연관된 기능을 구현
사용자 정보 입력을 위해 로그인 기능 작동을 확인 -> 이후에 정보 수집 화면으로 넘어감

팀원 B : '로그인 기능만 올려줘~'
하지만, 이미 정한 방식에 따르면 상위 브랜치는 하위 브랜치의 기능이 완료된 이후에 develop으로 합치기로 함

팀원 B는 로그인 작동에 대한 테스트를 못한 채로 개발을 진행해야함..

이렇게 된다면 연관된 기능이 있을 때마다 각 팀원들의 개발 과정이 늦춰지게 되는 것이다.

해당 이슈를 접하게 되면서 브랜치를 세분화하는 것이 무조건적으로 옳은 게 아니란 것을 깨닫고 해당 방식에 대해서 좀 더 많은 시나리오를 세우지 못한 것을 반성했다.


이를 해결하기 위해선 2가지 정도의 대안책을 생각했다.

  1. 세분화를 포기하고 develop 브랜치에 지속적으로 합치기

  2. 연관된 기능에 대해선 상위 브랜치를 공유하기

사실 2번째 방법을 처음부터 상정하고 좀 더 치밀하게 전략을 세웠어야 했는데 급작스럽게 이슈를 맞이하면서 1번째 방법으로 개발을 진행하게 됐다.

물론 이 방법이 잘못된 방법은 아니라고 생각하지만, 다음 협업을 진행할 땐 개발 시나리오에 대한 내용을 좀 더 찾고 프로젝트의 흐름과 전략이 잘 맞물리도록 설정해볼 생각이다.

RESTful API란?
REST (Representational State Transfer)는 웹 애플리케이션 설계 아키텍처 중 하나로, 리소스를 URI로 표현하고, HTTP 메서드를 통해 해당 리소스를 조작하는 방식입니다. RESTful API는 이 REST 원칙을 따르는 API입니다.
